# Leonardo — What works in Beta 0.1

_Audit date: 2026-06-14. Method: live probes against the running stack + automated tests + source review. Every "WORKS" below was reproduced against the live services, not inferred from code._

Update note: closed-beta Imagination Graph/Council Memory MCP developer access is now a separate read-only surface. Treat it as independent of the more complete Agent Trust Stack; it does not by itself include Council/Workshop intake, receipts, gates, token rails, or verdict authority.

## TL;DR

Beta 0.1 is a **read-only Leonardo chatbot** with exactly two powers, both verified live:

1. **The imagination graph** — search 577K concepts and pull each one's real source (author, work, year, passage), its neighbors, and its Bible parallels.
2. **Council memory** — recall what the Council has actually deliberated and ruled.

That is the whole product surface. No paid actions, no Workshop, no on-chain writes are reachable by the chatbot. Payments are off (`METER=false`).

## Architecture (the path a message actually takes)

```
Browser (Vercel, wallet-gated)
  → gateway  /api/chat        (Hono on bun, :8787, Bearer-token locked)
    → Hermes ACP bridge       (services/hermes-acp/bridge.py — one long-lived read-only agent)
      → leonardo-graph MCP    (services/graph-mcp/server.py — 5 read-only tools)
        → Neo4j :7687         (imagination graph)
        → workshop-sidecar    (:8799 → council memory graph :7688, bible/related/concept)
```

Active chat brain is `CHAT_BRAIN=hermes` — the real read-only Leonardo agent, not the fallback codex/anthropic path. It runs its own tool loop, denies every tool permission (no writes, no shell, no filesystem), and streams its answer over SSE.

## Access — yes, a closed-beta user can join right now

The join flow is live and gated end-to-end:

1. Every page is behind `middleware.ts` → unauthenticated visitors are sent to `/gate`.
2. `/gate`: connect wallet → sign a fresh timestamped message → `POST /api/auth`.
3. `/api/auth` verifies the signature (viem), enforces a 5-minute freshness window, and checks
   **`ALLOWED_WALLETS`** — the closed-beta lock (set during private beta; unsetting it is the public-launch switch).
4. On success it issues a 14-day HMAC session cookie; the same token rides every gateway call as `x-leo-session`.

So only allowlisted wallets can get in, and they prove ownership by signature — no passwords to leak.

## He knows them by user — verified live

The gateway verifies the `x-leo-session` HMAC (shared `SESSION_SECRET`) → resolves the wallet → injects a
read-only **MEMBER CONTEXT** preamble into a fresh ACP session: *"recognized Leonardo beta member, wallet 0x…,
member since DATE, conversation #N,"* plus summaries of **their own** past conversations. Strict per-wallet
scoping — he is told never to reveal another member exists.

Live proof (forged a valid member token, fresh conversation):
- Leonardo answered: *"Yes—welcome, Leonardo beta member; I can help you mine the imagination graph, recall
  prior Council rulings, and ground concepts in provenance, parallels, and source text."*
- A per-user identity file was created at `~/.leonardo-platform/identities/<wallet>.json`
  (`first_seen`, `last_seen`, `visits`, and a per-conversation summary) — so recognition **persists across
  visits and gateway restarts**, and the next visit recalls the gist of this one.

## Explicitly NOT in v0.1 (exists in the repo, not reachable by the chatbot)

- **Workshop research briefs** — `workshop_research` is **not** in the MCP tool list (`graph-mcp/server.py` exposes 5 tools only), so the web agent cannot call it. The `/research` sidecar endpoint exists but is unwired from chat.
- **Paid Council (quick $0.05 / panel $0.25)** — `/api/council/review` and `/api/council/panel` exist on the gateway but are **not** MCP tools for the chatbot, and `METER=false` means no x402 charge path is active.
- **x402 pay-as-you-go paywall** — off (`METER=false`); the "first 5 free then pay" flow is dormant in this deploy.
- **ERC-8004 on-chain / Repro Lab evals** — present in the codebase, not part of the chat experience.

## Honest qualifications (from independent adversarial re-verification, 8 agents)

All 10 capabilities above were re-probed and confirmed. These are the precise caveats a truthful v0.1 readout should carry:

- **The whole v0.1 shape is config-dependent on three env values** (all set correctly right now): `CHAT_BRAIN=hermes`, `METER=false`, `GATEWAY_TOKEN` set. If the brain were switched to `codex/openai/anthropic`, the alternate `runChatTurn` path *does* expose `workshop_research` + paid Council to the model (via `chat/tools.ts`); if `METER=true`, the x402 paywall activates; if `GATEWAY_TOKEN` were unset, the gateway would be open. So v0.1's safety/scope is a property of this deployment's configuration, not an absolute code-level removal.
- **Council memory uses the real semantic graph, not the file log.** With the hermes brain, `council_memory` → sidecar `/council/search` → Council Neo4j (:7688) vector search (returns dated, seat-attributed, VERIFIED/CONTESTED verdicts; ranking varies correctly by query). A *separate* file log (`~/.leonardo-platform/council-memory/log.json`) holds only mock entries and is read **only** by the inactive non-hermes path — so it never reaches the live chatbot.
- **Web search is now ENABLED (2026-06-14).** The free `ddgs` (DuckDuckGo) backend was installed into the shared hermes venv and the `leonardo-web` profile was pointed at it (`web.backend`/`web.search_backend: ddgs`) with the `environment_hint` updated to permit `web_search`. Live-verified: a "current population of Tokyo" turn ran 3 web searches + 1 page extract and cited the official Tokyo Metropolitan Government URL. Costs nothing (no API key); to upgrade quality later, add a paid key (`TAVILY_API_KEY`/`EXA_API_KEY`/`FIRECRAWL_API_KEY`) to `~/.hermes/.env`. Note: because `ddgs` is now importable in the shared venv, the other fleet profiles (leonardo/gabriel/castor) will also auto-detect it as their web backend on next restart.
- **Provenance `year` can be malformed for some concepts** (e.g. a `time travel` mention came back `year=101`). The cited *Shaman (2013)* is clean, but don't treat every `year` as reliable.
- **Honest-failure, not fabrication:** every tool returns `{ok:false,error:...}` when its backend is down. Citations depend on the sidecar + Neo4j being up (they are).
- **Test counts are network-sensitive:** the stable result is 111 pass / 3 fail (the `gateway-evals` `localStorage` env bug). A couple of `*.live.test.ts` files hit Base RPC and can flake the count upward if the RPC is unreachable.

## One honesty caveat for the UI copy

The chatbot is grounded and read-only, but it still runs on an LLM substrate — it is not cryptographically authenticated, not infallible, and not production-authorized. The landing copy now states only the two real capabilities (graph + council memory) and no longer promises the Workshop or paid Council panels.
